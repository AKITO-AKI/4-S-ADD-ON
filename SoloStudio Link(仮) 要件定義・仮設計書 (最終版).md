# **三年次課題研究**

## **完全分業型V2Vコンポジットアドオン「SoloStudio Link(仮)」要件定義・仮設計書**

### **1\. プロジェクトの目的と基本思想**

* **基本思想:** 「人間の作業（演技・カメラワーク）」と「AIの作業（背景・照明・ルック開発）」を完全に分業する「一人アニメスタジオ」の構築。本ツールは開発者自身のローカル環境における映像制作プロセスを極限まで効率化するための専用ツールとして位置づける。  
* **目的:** Blender上で作成した簡素な3Dアニメーション（キャラクターの動きとカメラワークのみ）から、ローカルAIのV2V (Video to Video) 技術を用いて、背景・照明・質感が完全に統合された高品質なアニメーション映像を自動生成するアドオンを開発する。

### **2\. システム・アーキテクチャ(全体構成)**

フロントエンド（操作画面）とバックエンド（AI処理）を完全に分離し、API通信で繋ぎます。

| 役割 | ソフトウェア/技術 | 詳細 |
| :---- | :---- | :---- |
| **フロントエンド** (UI / データ抽出) | Blender (Python API) | ユーザー操作、3Dモーションの作成、参照用連番画像の高速・正確な出力（Workbench/コンポジットノード使用）。 |
| **バックエンド** (AI処理) | ComfyUI (Local API) | ノードベースの画像/動画生成環境。BlenderからのAPIリクエストを受け取り、バックグラウンドでV2V処理を実行。 |
| **通信プロトコル** | HTTP (REST) / WebSocket | Blenderからの処理開始指示（POST）に加え、AI処理の進捗状況を受信するためにWebSocketを利用。 |

### **3\. 使用AIモデルの選定**

AIの推論結果を安定させ、フリッカー（チラつき）やデザインの崩れを防ぐための構成。ローカルの高性能マシンのリソースを最大限活用する。

1. **ベースモデル:** Animagine XL 3.1 (または同等のSDXLアニメ特化モデル)  
   * 理由: アニメ調ルックにおいて最もプロンプトへの追従性が高く、背景の描き込みが美麗なため。  
2. **動画生成モジュール:** AnimateDiff for SDXL  
   * 理由: 複数枚の画像を連続した映像としてAIに認識させ、背景や光のチラつきを抑え込むための心臓部。  
3. **制御モジュール1:** ControlNet-XL Depth  
   * 理由: カメラのパンニングやキャラクターの前後関係といった空間情報を絶対のルールとしてAIに守らせる。  
4. **制御モジュール2:** ControlNet-XL Lineart (または Canny)  
   * 理由: アニメーターが手付けした細かな指の動きやポージング（シルエット）をAIの幻覚から守る。  
5. **キャラデザイン固定モジュール:** IP-Adapter (Plus / FaceID)  
   * 理由: **\[重要\]** プロンプトに依存せず、キャラクターの服の色、髪型、瞳のハイライトなどを動画全体で一貫させるため。

### **4\. データ・パイプライン(処理の流れ)**

アドオンの「実行」ボタンを押下後、以下のフローが全自動で進行する。

**【Step 1: Blenderからのデータ抽出】**

Blenderのバックグラウンド処理で、タイムラインから以下の連番画像を出力する。

* depth\_xxx.png (Z深度マップ)  
* lineart\_xxx.png (輪郭線パス)  
* char\_ref.png (キャラクター設定画: ユーザー指定)

**【Step 2: APIリクエストと非同期通信開始】**

抽出したデータとユーザー入力をJSON化し、ComfyUIへ送信する。

同時に、BlenderのPythonスクリプトはWebSocketによるリスナーを立ち上げ、ComfyUIからのレスポンス待機状態に入る。

**【Step 3: ComfyUIでのV2V生成】**

ComfyUI側で「AnimateDiff \+ ControlNet \+ IP-Adapter」が稼働。長尺対応のため、AnimateDiffの設定で「Sliding Window Context」を利用し、一定フレームごとに文脈をずらしながら映像を生成する。

**【Step 4: 進捗表示とフィードバック】**

生成完了後、生成された動画（.mp4 または 連番画像）をBlenderの「ビデオシーケンサー(VSE)」に自動ロードし、即座にプレビュー可能にする。

### **5\. Blenderアドオン UI (操作画面)の仮設計**

Nパネル（画面右側のサイドバー）に配置する、作業効率を最優先したシンプルなUI。

* **パネル名:** SoloStudio Director  
* **入力項目:**  
  * **Prompt:** 描きたい背景や光の指示  
  * **Negative Prompt:** 回避したい要素  
  * **Character Reference:** キャラクターの見た目を固定するための設定画像のパス指定  
  * **Frame Range:** 開始フレーム / 終了フレーム  
* **情報表示:**  
  * **Progress Bar:** 生成の進捗状況（%）と、現在の処理ステータス（「データ抽出中」「AIレンダリング中」等）を表示。  
* **実行ボタン:** \[Render AI Video\]

### **6\. 開発における懸念事項とリスク対策 (Risk Management)**

本システムの確実な完成に向け、実装フェーズで直面する可能性が高い技術的課題と、その解決策を定義する。

#### **1\. UXの崩壊（API通信時のBlenderフリーズ問題）**

* **懸念:** 動画生成（数十秒〜数十分）の間、Pythonの単純な同期処理（requests.postなど）で応答を待機すると、Blenderのメインスレッドがブロックされ「応答なし（フリーズ）」状態に陥る。これにより、ユーザーが強制終了してしまうリスクがある。  
* **対策:** Blenderアドオンの実装において、非同期通信（asyncio）やBlender内蔵のモーダルタイマーオペレーター（bpy.app.timers または modal）を使用する。バックグラウンドでComfyUIのWebSocketサーバーと通信し、BlenderのUIの再描画を止めずにプログレスバーを更新し続ける設計とする。

#### **2\. AIモデル同士の干渉（パラメータ調整の沼）**

* **懸念:** AnimateDiff（動き）、ControlNet（形）、IP-Adapter（キャラ）を同時に稼働させると、それぞれの指示が衝突し映像が破綻しやすい。特にIP-Adapterの強度が強すぎると、プロンプトで指示した環境光（夕焼けなど）が無視され、ControlNetが強すぎると動きが固くなる現象が発生する。  
* **対策:** アドオンのコーディングに入る前に、ComfyUIのノードエディタ単体で「各モデルの最適な強度バランス（黄金比）」を見つけるPoC（概念実証）期間を設ける。見つけ出した最適なパラメータはJSONファイル内に固定値として埋め込み、BlenderのアドオンUI上ではユーザーに複雑な数値を操作させない（ブラックボックス化する）ことで、常に出力が安定するツールにする。
